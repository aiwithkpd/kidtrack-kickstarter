Overview

What we're building and why

The KidTrack hardware watch tracks a toddler's GPS position, activity, and sleep via an nRF52840 MCU, MAX-M8Q GPS module, and LSM6DS3TR-C 6-axis IMU. The companion app is its brain extension — turning raw BLE telemetry into actionable parenting insights, with no cloud dependency and no subscription.

Problem

Existing child tracker apps require expensive monthly subscriptions, rely on cellular infrastructure, and are built for older children. Parents of toddlers (1–4 years) have no lightweight, offline-capable, open option for proximity monitoring and developmental tracking.

Solution

A BLE-native mobile app (iOS + Android) that pairs directly with the KidTrack watch, visualizes location on offline maps, surfaces activity and sleep analytics, and provides a fully open developer mode — all without a server or monthly fee.

ℹ️
BLE-first architecture
The app communicates exclusively over Bluetooth Low Energy in v1.0. GPS data, IMU telemetry, and battery state are all streamed via the nRF52840's GATT server. Cellular / LTE-M connectivity is a planned stretch goal and will be introduced as an optional backend module in a future release.

Scope

In scope vs. out of scope

Version 1.0 is intentionally scoped to BLE-only functionality. Cellular and cloud features are deferred.

✓ In Scope — v1.0
  • BLE pairing and live GATT data streaming
  • GPS location display on offline maps
  • BLE proximity alerts (virtual tether)
  • Last-known location pin on BLE drop
  • IMU activity classification display
  • Sleep detection and nap history calendar
  • Battery % display + estimated runtime
  • GPS polling rate toggle (slider)
  • Raw telemetry dev console mode
  • OTA firmware update over BLE
  • Data export (CSV, GPX, JSON)
  • Multi-child device switching
  • Local co-parent BLE auth handoff
✕ Out of Scope — v1.0
  • Cellular / LTE-M tracking (stretch goal)
  • Cloud storage or server sync
  • Third-party account login (Google, Apple)
  • Video monitoring or camera integration
  • AI / ML insights (deferred to v2)
  • Wearable heart rate monitoring
  • Social or sharing features
  • In-app purchases or premium tiers
  • Web dashboard / browser app
  • Watch face customization UI
Feature Requirements

All 6 feature domains

Requirements are tagged P0 (must-have launch blocker), P1 (high priority), P2 (important), or P3 (nice-to-have).

Domain 1 of 6
Real-Time Tracking & Mapping
Core navigation — proximity view, offline maps, last-known position, and dead-reckoning
  • Proximity Map View
    Live map showing parent phone location relative to child watch position. Updates on each BLE GATT notification from the GPS characteristic. Displays distance in meters and cardinal direction.
    P0
  • Offline Map Tiles & Caching
    Allow pre-downloading local map regions (e.g., home neighborhood, local park). Required because BLE-only operation means no guaranteed internet access during use. Cache via MBTiles or equivalent local tile storage.
    P0
  • Last-Known Location (LKL) Pin
    On BLE signal loss, immediately drop a persistent pin on the map at the final confirmed GPS coordinate received. Pin shows timestamp and accuracy radius. Tap for address reverse-geocode (if online).
    P0
  • IMU Dead-Reckoning Visualizer
    When GPS lock is lost indoors, use streamed X/Y/Z accelerometer + gyroscope data to render an estimated movement path on the map. Display as a dotted arrow vector. Confidence degrades over time without GPS fix.
    P2
Domain 2 of 6
Smart Alerts & Safety
Proactive monitoring — virtual tether, reconnection alerts, tamper detection
  • Virtual BLE Tether (Geofence)
    Adjustable proximity alert at 10m, 20m, or 30m based on BLE RSSI signal strength drop. When threshold is crossed, trigger a loud distinct alarm on the parent phone (overrides silent mode). User-configurable threshold and alarm tone.
    P0
  • Reconnection Alert
    Subtle, reassuring haptic + notification when child comes back into BLE range after a drop. Distinct from the tether alarm. Shows how long the connection was lost and LKL summary.
    P1
  • Tamper / Removal Detection
    Monitor IMU for sudden large-magnitude gyroscope spikes inconsistent with normal toddler movement (e.g., rapid rotation + immediate stillness). Alert parent when watch is potentially removed or forcibly pulled off.
    P2
Domain 3 of 6
Health, Activity & Sleep Analytics
IMU-driven metrics — step counter, activity rings, automated sleep logging, wake alerts
  • Activity Metrics Dashboard
    Daily summary screen showing: step-equivalent count (toddler shuffle translated to steps), active play vs. stillness time as a ring/pie chart, activity level timeline (sparkline per hour). All derived from on-device IMU classification.
    P1
  • Automated Sleep & Nap Logger
    Since the watch auto-detects sleep without button input, display a Sleep History Calendar showing: overnight sleep start/end, nap periods (morning vs. afternoon), sleep quality score based on micro-movement restlessness index.
    P1
  • "Toddler Awake" Push Notification
    Fire a quiet/silent push notification the moment the IMU registers wakeful movement patterns after a recorded nap. Configurable per nap session — parents can disable for overnight sleep to avoid disturbance.
    P1
Domain 4 of 6
Device Management & Power Metrics
Battery display, estimated runtime, GPS polling toggle, low-battery alerts
  • Dynamic Battery Life Indicator
    Show exact battery percentage from the A0 analog pin telemetry streamed over BLE battery level characteristic. Display as both percentage and a visual gauge. Update in real time on each BLE notification.
    P0
  • Battery-to-Time Calculator
    Display estimated remaining runtime in hours based on current GPS polling interval and historical drain rate. Formula: remaining_mAh ÷ current_draw_mA = hours. Update dynamically when user changes GPS mode.
    P1
  • Adaptive GPS Polling Toggle
    A slider with two named modes: High-Security (5s interval, ~18–24hr battery, recommended for crowded public spaces) and Battery-Saver (30s interval, ~3–4 day battery, recommended for home use). Writes new poll interval to watch via BLE write characteristic.
    P1
  • Low Battery Warning System
    Send a critical push notification when battery drops to 20% (caution) and again at 10% (critical). Notification text distinguishes urgency: "KidTrack is low — charge soon" vs. "KidTrack critically low — charge now." Persistent until dismissed.
    P0
Domain 5 of 6
Open-Source & Developer-Friendly Features
Raw telemetry console, OTA firmware updates, data export in open formats
  • Raw Telemetry Stream (Dev Mode)
    Toggleable console view showing in real time: raw NMEA GPS sentences, X/Y/Z accelerometer values (m/s²), X/Y/Z gyroscope values (dps), BLE RSSI (dBm), and MCU-reported activity state. Scrollable log with timestamp. Exportable as plain text.
    P2
  • OTA Firmware Updates via BLE
    Implement Nordic DFU (Device Firmware Upgrade) over BLE to allow flashing updated or community firmware .zip packages directly from the app onto the nRF52840. Show progress bar and checksum verification. Warn user not to close app or leave BLE range during flash.
    P2
  • Data Export
    Export all locally stored data in three open formats: .CSV (spreadsheet-compatible activity + sleep log), .GPX (GPS track for Google Earth / mapping tools), .JSON (full structured data dump for developers). Share via system share sheet.
    P1
Domain 6 of 6
Multi-Device & Profile Management
Multi-child dashboard, co-parent local BLE auth handoff
  • Multi-Child Dashboard
    Allow pairing and simultaneous monitoring of multiple KidTrack watches (one per child). Each child has a named profile with an avatar and color. Dashboard shows all children's status at a glance. Switch between individual detail views with a tab or swipe.
    P2
  • Co-Parent Local Link (BLE Auth Handoff)
    Securely authorize a second phone (co-parent or babysitter) to pair with the watch's BLE profile using a one-time passkey shown on the primary device. Whichever authorized phone is closest becomes the active tracking terminal. No server required — handoff is local BLE negotiation only.
    P2

Technical Architecture

System layers

The app is a thin BLE client. All sensor processing happens on-device in the watch firmware.

UI Layer — React Native (iOS + Android)
React Native 0.74+ react-native-maps Victory Native (charts) react-navigation Zustand (state)
BLE Communication Layer
react-native-ble-plx BLE 5.0 GATT Client Nordic DFU (OTA) Custom GATT UUIDs RSSI monitoring
Local Storage Layer
SQLite (location history) AsyncStorage (settings) MBTiles (offline maps) MMKV (fast KV cache)
Hardware Interface (Watch Firmware — nRF52840)
GPS GATT Characteristic IMU GATT Characteristic Battery Level Characteristic Activity State Characteristic Config Write Characteristic
⚠️
No backend in v1.0
All data stays on-device in local SQLite. The co-parent sync is a local BLE peer-to-peer handoff. Cloud backup is a future roadmap item (with explicit opt-in and end-to-end encryption when added).

Risks & Mitigations

What could go wrong

Key technical and product risks identified during scoping.

📡
BLE range is insufficient for real-world use
nRF52840 BLE range of ~30m may be too short in open spaces like large parks. RSSI-based distance is also inconsistent across phone hardware.
Mitigation: LKL pin covers the gap. Stretch goal adds cellular LTE-M. RSSI thresholds are user-adjustable.
High
🗺️
Offline maps are large and complex to cache
MBTiles for a city-sized region can be 500MB+. Storage and UX around downloading tile packs adds complexity and friction.
Mitigation: Limit initial tile cache to a small radius (e.g., 5km around home). Defer large region download to v1.1.
Medium
😴
Sleep detection false-positive rate is high for toddlers
Toddlers are restless sleepers. IMU-based sleep detection tuned for adults may generate frequent false wakes or miss nap start.
Mitigation: Use beta backer feedback to tune thresholds. Provide manual override / correction UI. Ship with conservative (less sensitive) defaults.
Medium
📱
iOS BLE background scanning limitations
iOS restricts background BLE scanning. The proximity alarm may not fire if the app is fully backgrounded and the phone is locked on iOS 17+.
Mitigation: Use Core Bluetooth's state restoration + background modes. Display prominent "keep screen on" guidance for high-security use cases.
High
🔒
Co-parent BLE handoff is spoofable
A local BLE passkey system without a server-side trust anchor could be vulnerable to nearby spoofing attacks.
Mitigation: Use BLE Secure Simple Pairing (SSP) with numeric comparison. Require physical proximity (<1m) during handoff. Document security model openly.
Low
